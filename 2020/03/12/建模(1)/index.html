太极建模口诀:由外向内，层次分明，动静结合，逐步求精

由外向内：(系统边界)对于系统而言，用户就是系统的外部，由外部用户出发向内开发。

层次分明：(用户目标，真正的用例)可以把同一层面的问题，分解成不同层面的问题，从数量上降下来。

动静结合:(fr(用例)/uc + nfr)多种方面。程序执行是动，程序的属性就是静

逐步求精:(先图形，后文本)前期实现一个核心原型，然后逐步优化，不断去得到一个最优化的系统。

uml静态建模

1.理解用例模型

2.3.4为core

2.识别分析类: 识别实体类，识别边界类，识别控制类

实体类: 表示系统存储和管理的永久信息(数据)

边界类: 表示参与者与系统之间的交互(界面)

控制类: 表示系统在运行过程中的业务控制逻辑

3.定义交互行为

4.建立分析类图: 定义属性，定义行为，定义关系

5.评审分析模型

一般来说，一个用例对应一个控制类

面向对象模型更接近于问题域，以问题域中的对象为基础建模

面向对象分析(object oriented analysis ooa)

面向对象设计(object oriented design food)

面向对象编程(object oriented programming pop)

面向对象测试(object oriented testing oot)

uml(unified modeling language):

统一建模语言是一种直观化，明确化，构建和文档化软件系统产物的通用可视化建模语言。

uml代替了传统的“边想边写”开发方式

uml可以描述开发所需要的各种视图:

概念的: 业务过程和系统功能

具体的:程序中的类，数据库模式，可重用的软件构件

uml基本构造块:

事物:表示系统中的元素，包括:结构事物，行为事物，分组事物，注释事物

关系:表示系统中的元素如何进行连接；包括:依赖，关联，泛化，实现

图:系统模型中每一个视图的内容是由一些图来描述的；对整个系统而言，其功能由用例图描述，静态结构由类图和对象图描述，动态行为由状态图，时序图，协作图和活动图描述，而物理架构则是由组件图和部署图描述。

用例图:

用例是从用户角度描述系统的行为

用例之间的关系:

关联(association):参与者与用例间的关系;

【箭头指向】：指向消息接收方

包含关系(include):用例之间的关系

【箭头指向】：指向分解出来的功能用例

扩展关系(extend):用例之间的关系

【箭头指向】：指向基础用例

泛化关系(generalization):参与者之间或用例之间的关系

【箭头指向】：指向父用例
